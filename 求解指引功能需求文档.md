# 求解指引功能需求文档

> 版本：v1.1 | 状态：已确认，待实施
> 功能代号：Hint System

---

## 一、功能概述

在每轮游戏结束后，玩家可通过指令查询指定题目的**锁格推理路径**。功能目标是展示从当前盘面出发，通过最精简的逻辑步骤推导出目标格子应填数字的完整过程，帮助玩家理解解题逻辑，而非单纯公布答案数字。

**约束**：求解器**不允许使用回溯/试探法**，只能使用纯逻辑推理技巧。若所有支持的技巧均无法锁定目标格，则提示「此题超出当前支持的逻辑技巧范围，无法生成推理路径」。

## 二、题目编号机制

### 2.1 编号结构

每道题目的编号由 **"轮次前缀 + 题序数字"** 组成：

```
a1, a2 … a8    第1轮，共8题
b1, b2 … b8    第2轮
…
z1, z2 … z8    第26轮
aa1, aa2 … aa8 第27轮，前缀进位为两位
ab1 … az8      第28~52轮
ba1 … zz8      进一步进位，共 26×26=676 轮
aaa1 …         前缀满两位后进三位，以此类推
```

### 2.2 进位规则

- 前缀为纯小写字母，规则等价于 **26 进制计数**
- 第 1 轮 = `a`，第 26 轮 = `z`，第 27 轮 = `aa`，第 52 轮 = `az`，第 53 轮 = `ba`，以此类推

### 2.3 编号作用域

- 编号**按频道（群）独立维护**，A群的 `a` 轮与 B群的 `a` 轮互不干扰
- 每个频道独立维护当前轮次计数器及缓存数据

### 2.4 24 小时回收与前缀分配规则

- 每个前缀记录其**首次使用时刻（Unix 时间戳）**
- 超过 **24 小时**后，该前缀及其题目数据自动释放，前缀重新可用
- **每次新游戏开始时，必须分配一个新前缀**，绝不复用当前仍在缓存中的前缀
- **前缀分配优先级：优先复用已过期（回收）的最小前缀，若无可回收前缀则顺序递增到新前缀**

**示例：**
```
当前已用前缀：a（已过期），b（已过期），c（活跃），d（活跃）
→ 下次新游戏分配前缀：a（最小已过期的前缀优先）
→ 再下次：b
→ 再下次：e（c/d 未过期，跳过，取下一个新前缀）
```

## 三、数据缓存设计

### 3.1 缓存位置

- **内存缓存**（进程重启后数据清空，属正常行为）

### 3.2 缓存结构

每道题目缓存以下数据，保留 24 小时：

| 字段 | 类型 | 说明 |
|---|---|---|
| `questionId` | `string` | 题目编号，如 `a3` |
| `channelId` | `string` | 所属频道 ID |
| `puzzle` | `number[][]` | 出题时的 9×9 盘面矩阵（0 表示空格） |
| `solution` | `number[][]` | 完整解答矩阵 |
| `targetRow` | `number` | 问题格子行索引（0-based） |
| `targetCol` | `number` | 问题格子列索引（0-based） |
| `targetAnswer` | `number` | 该格应填数字 |
| `createdAt` | `number` | 创建时间戳（毫秒），用于判断是否过期 |

### 3.3 缓存 Map 键名格式

```
"<channelId>:<题目编号>"
示例: "701118454:a3"
```

### 3.4 前缀注册表结构

```typescript
// 每个频道维护所有"已知前缀"及其创建时间
type PrefixEntry = {
  prefix: string;
  createdAt: number; // 该前缀被分配时的时间戳（毫秒）
};
channelPrefixes: Map<string, PrefixEntry[]>
// key = channelId，value = 该频道历史上所有使用过的前缀列表
```

## 五、指令规格

### 5.1 指令名称

- 指令名可在插件配置中设置，**默认为 `获取答案`**
- 完整调用格式：`获取答案 <题目编号>`

### 5.2 调用示例

```
获取答案 a1     查询本群 a 轮第 1 题的推理路径
获取答案 ab3    查询 ab 轮第 3 题
```

### 5.3 响应规则

| 情况 | 机器人回应 |
|---|---|
| 编号有效且未过期，且该轮已结束 | 输出推理路径文本 |
| 编号属于**当前正在进行的轮次** | 「本轮游戏尚未结束，无法查询当前题目答案」 |
| 编号不存在（从未出现过） | 「题目编号无效，请确认编号是否正确」 |
| 编号已过期（超 24 小时后被回收） | 「该题目已过期，无法查询」 |
| 技巧范围不足以推导 | 「此题超出当前支持的逻辑技巧范围，无法生成推理路径」 |

> **注意**：游戏进行中，玩家仍可查询**之前轮次**的题目答案；被禁止查询的仅是**当前轮次**的题目。

## 六、求解路径内容设计

### 6.1 目标与约束

以**目标格**为终点，展示从初始盘面出发，通过逐步逻辑推理最终锁定该格数字的完整路径。

- **不允许使用回溯 / 试探法**，只能使用纯逻辑推理
- 优先使用最低层级技巧；若低层级无法推进，逐步尝试更高层级
- **无论使用哪个层级，只要能推导出答案，必须输出完整推理路径**
- 若直到 L5 仍无法锁定目标格，提示「超出支持范围」

### 6.2 推理技巧分级体系

| 层级 | 技巧名称 | 说明 |
|---|---|---|
| **L1** | **宫排除（Box Elimination）** | 目标格所在 3×3 宫中已有的数字，直接排除 |
| **L1** | **隐性唯余（Hidden Single）** | 某数字在某行/列/宫中只有唯一一格可填，直接确认 |
| **L2** | **行排除（Row Elimination）** | 目标格所在行已有的数字，直接排除 |
| **L2** | **列排除（Column Elimination）** | 目标格所在列已有的数字，直接排除 |
| **L2** | **区块排除（Pointing Pairs/Triples）** | 某宫中某候选数只分布于同一行/列，消除该行/列宫外其他格的该候选数 |
| **L3** | **显性唯余（Naked Single）** | 某格经三重排除后只剩 1 个候选数，直接确认 |
| **L3** | **显性数对/数组（Naked Pair/Triple）** | 若干格的候选数集合完全相同，消除同行/列/宫其他格的相关候选数 |
| **L3** | **隐性数对/数组（Hidden Pair/Triple）** | 某行/列/宫中若干候选数只出现在相同的若干格中，消除这些格的其他候选数 |
| **L4** | **区块唯余组合** | 区块排除与唯余法的联合推理 |
| **L4** | **数组唯余组合** | 数对/数组与唯余法的联合推理 |
| **L5** | **多重复杂组合** | 多个 L3/L4 技巧交叉组合，适用于高难度题目 |

> **重要**：
> - 每一步推理只使用当前层级可完成的操作
> - 若推进某步需要同时依赖 L1 和 L2，则该步归类为 L2
> - 整体推理路径中最高使用层级即为该题的「求解难度评估」

### 6.3 路径输出格式

```
【A6 格推理路径】目标答案：7

步骤 1 → 宫排除（第7宫）[L1]
  第7宫已有数字：1, 3, 4, 6, 8
  A6 候选数剩余：2, 5, 7, 9

步骤 2 → 行排除（第A行）[L2]
  第A行已有数字：1, 3, 4, 5, 6, 8, 9
  A6 候选数剩余：2, 7

步骤 3 → 列排除（第6列）[L2]
  第6列已有数字：1, 2, 3, 4, 5, 6, 8
  A6 候选数剩余：7

步骤 4 → 显性唯余 [L3]
  A6 仅剩候选数 7，直接确认 ✓

使用技巧：宫排除 → 行排除 → 列排除 → 显性唯余
最高层级：L3（中级）
```

### 6.4 求解难度与层级对应

| 层级上限 | 难度描述 |
|---|---|
| L1 | 入门 |
| L2 | 基础 |
| L3 | 中级 |
| L4 | 进阶 |
| L5 | 高阶 |
| 超出范围 | 无法生成推理路径 |

## 七、技术实现概要

### 7.1 前缀分配器（内存）

```typescript
type PrefixEntry = {
  prefix: string;
  createdAt: number;
};
// key = channelId
channelPrefixes: Map<string, PrefixEntry[]>
```

**分配算法（每次新游戏开始时调用）：**
1. 取出该频道的 `PrefixEntry[]`（没有则初始化为空）
2. 收集所有 `createdAt` 超过 24 小时的条目，视为"可回收前缀"
3. 若有可回收前缀，按字典序排序，取最小的一个前缀复用（同时更新其 `createdAt` 为当前时间）
4. 若无可回收前缀，则基于"现有最大前缀"递增得到新前缀，追加到列表中
5. 返回本轮使用的前缀

### 7.2 问题数据缓存

```typescript
type QuestionRecord = {
  puzzle: number[][];
  solution: number[][];
  targetRow: number;
  targetCol: number;
  targetAnswer: number;
  createdAt: number;
};
// key 格式: "<channelId>:<questionId>"
questionCache: Map<string, QuestionRecord>
```

查询时懒惰清理：若访问某 key 发现已超 24 小时，直接视为「已过期」并删除。

### 7.3 求解算法流程

```
初始化候选数矩阵（candidates[9][9]）
  → 每格候选数 = {1..9} 减去行/列/宫已有数字

循环（直到目标格确定 或 无法继续推进）：
  尝试 L1 技巧（宫排除、隐性唯余）
  → 若有任意格候选数减少，记录步骤，继续循环
  尝试 L2 技巧（行排除、列排除、区块排除）
  → 若有任意格候选数减少，记录步骤，继续循环
  尝试 L3 技巧（显性唯余、Naked/Hidden Pair/Triple）
  → …
  尝试 L4 技巧
  → …
  尝试 L5 技巧
  → …
  若所有技巧均无法减少任意候选数：退出循环，标记「超出范围」

目标格候选数 == 1 → 推理成功，输出路径
```

### 7.4 修改涉及的文件

| 文件 | 变更内容 |
|---|---|
| `src/hint.ts`（新建） | 前缀分配器、`QuestionRecord` 缓存管理、求解算法 |
| `src/renderer.ts` | 渲染底部区域：左/中显示 `level N`，右侧显示题目编号 |
| `src/game.ts` | `askNextQuestion` 中注册题目到缓存；新增 `showHint` 命令处理 |
| `src/index.ts` | 注册 `commandHint` 配置项和对应命令 |
